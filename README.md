# AVT — Auto V Korea

`AVT — Auto V Korea` — это полнофункциональная веб-платформа для продажи и подбора автомобилей из Южной Кореи с интеграцией с площадкой `Encar`.

Проект совмещает в себе:
- публичный сайт с каталогом автомобилей;
- детальные карточки машин с расчётом итоговой стоимости;
- административную панель;
- встроенный парсер Encar;
- систему хранения, нормализации и обогащения данных;
- расчёт цен в KRW и USD по живому курсу;
- поддержку инспекционных отчётов Encar.

Проект ориентирован не просто на витрину объявлений, а на рабочий инструмент для посредников и команд, которые регулярно подбирают и выкупают автомобили в Корее.

## Что представляет собой проект

Сайт решает сразу несколько задач:

1. Собирает объявления с Encar и сохраняет их в локальную базу.
2. Приводит разнородные корейские данные к единому русскоязычному виду.
3. Показывает клиенту каталог с фильтрами, поиском, ценой и диагностикой.
4. Дает менеджеру админку для ручного управления каталогом и парсером.
5. Автоматически пересчитывает стоимость автомобиля с учетом:
   - курса USD/KRW;
   - фиксированного смещения курса сайта;
   - комиссии;
   - доставки;
   - погрузки / выгрузки;
   - стоянки;
   - возврата НДС.

Иными словами, это не просто "React-сайт", а прикладная система вокруг корейского авторынка.

## Основные возможности

### Публичная часть

- главная страница с презентацией услуги;
- каталог автомобилей;
- карточки автомобилей с фото, цветами, приводом, комплектацией, ключами, объёмом двигателя и местоположением;
- подробная страница автомобиля;
- отображение инспекционного отчёта Encar;

- мультифильтры по марке, топливу, приводу, типу кузова, цвету, году, цене и пробегу;
- сокращённые локации для быстрого просмотра;
- показ цены одновременно в KRW и USD.

### Админка

- вход по паролю;
- ручной запуск парсера;
- настройка расписания парсера;
- ограничение количества машин за один запуск;
- поток логов парсера в реальном времени;
- просмотр статистики;
- ручное создание и редактирование карточек;
- загрузка изображений;
- управление параметрами каталога.

### Серверная часть

- REST API для каталога;
- API-обогащение данных из Encar по `encar_id`;
- парсинг инспекционных отчётов;
- хранение машин, фото и конфигурации парсера в PostgreSQL;
- расчёт итоговой цены и возврата НДС;

## Что в проекте технически сложного

Ниже не просто список библиотек, а то, что реально делает проект сложным с инженерной точки зрения.

### 1. Гибридная интеграция с Encar

Проект работает не только с одним JSON API.

Он использует:
- прямой запрос к Encar API для карточки автомобиля;
- отдельный HTML-парсинг инспекционной страницы Encar;
- отдельный scraping-пайплайн для массового импорта каталога.

Это значит, что приложение совмещает:
- API integration;
- scraping;
- HTML DOM parsing;
- нормализацию данных из разных источников.

### 2. Парсинг корейских инспекционных отчётов

Инспекционные отчёты Encar не приходят в удобном готовом JSON для сайта.

Поэтому в проекте реализовано:
- получение HTML-страницы инспекции;
- декодирование корейской кодировки через `iconv-lite`;
- разбор структуры через `cheerio`;
- выделение:
  - summary-таблиц;
  - технических статусов;
  - истории ремонта;
  - фото инспекции;
  - комментариев инспектора;
  - подписей;
- перевод и нормализация корейских терминов в человекочитаемый русский вид.

Это одна из самых "нестандартных" частей системы.

### 3. Нормализация корейских автомобильных данных

Encar и корейские карточки часто отдают данные в неудобном виде:
- romanized Korean;
- смешанные англо-корейские названия;
- маркетинговые префиксы;
- неполные поля;
- строковые характеристики, зашитые прямо в названии.

В проекте есть отдельный слой нормализации, который:
- очищает шумные названия;
- убирает `The New`, `All New` и подобные префиксы;
- переводит нестандартные типы кузова;
- определяет привод из разных полей и из названия;
- вытаскивает безопасные комплектации из названия;
- нормализует цвета кузова и салона;
- приводит короткие локации к русскому виду;
- скрывает сомнительные значения, если данные ненадёжны.

Это по сути доменно-специфический ETL-слой прямо внутри приложения.

### 4. Двухэтапное обогащение каталога

Каталог не ограничивается тем, что было изначально сохранено в БД.

Если у машины:
- пустое название;
- нет привода;
- нет кузова;
- нет комплектации;
- нет ключей;
- нет объёма двигателя;
- слабые или отсутствующие фото;

клиентская часть может запросить доп. данные через `/api/encar/:encarId`, нормализовать их и затем отправить патч назад в локальную БД.

Это делает каталог "самообогащающимся" при просмотре.

### 5. Продвинутый расчёт цены

Ценообразование построено не на статической цифре.

В проекте реализовано:
- получение живого курса USD/KRW из интернета;
- fallback на резервный источник;
- кэш курса на сервере;
- правило сайта: `курс сайта = текущий курс - 15.0`;
- фиксированный возврат НДС `6%`;
- расчёт итоговой цены по составной формуле.

Это не просто отображение цены, а небольшой pricing-engine.

### 6. Реальное время в админке

Админка парсера использует `Server-Sent Events (SSE)`.

Это позволяет в реальном времени показывать:
- статус работы;
- прогресс;
- новые логи;
- завершение задачи;
- статистику по фото, ошибкам и пропускам.

То есть парсер и админ-панель связаны не polling-only моделью, а live-stream подходом.

### 7. Планировщик фоновых задач

Парсер можно запускать:
- вручную;
- каждый час;
- каждый день.

Расписание хранится в БД и восстанавливается при рестарте сервера.

Это означает, что система уже ближе к простому back-office сервису, чем к обычному сайту-визитке.

### 8. Динамические фильтры и поиск по каталогу

Фильтрация строится не на фиксированных константах, а на данных из БД и агрегатах.

Сервер генерирует:
- доступные бренды;
- типы топлива;
- приводы;
- кузова;
- цвета кузова и салона;
- диапазоны года, цены и пробега.

Плюс поиск работает не только по `name`, но и по:
- `model`
- `VIN`
- `encar_id`
- типу кузова
- приводу
- коробке
- цветам
- году
- объёму двигателя
- тегам

## Поток данных

### Сценарий массового импорта

1. Администратор запускает парсер.
2. Парсер обращается к Encar API.
3. Полученные записи нормализуются.
4. Рассчитываются коммерческие поля.
5. Фото проверяются и подгружаются.
6. Машины сохраняются в PostgreSQL.
7. Админка получает live-логи через SSE.

### Сценарий просмотра карточки

1. Пользователь открывает каталог или детальную страницу.
2. Клиент получает базовую карточку из `/api/cars`.
3. Если данных не хватает, фронт обращается к `/api/encar/:encarId`.
4. Сервер подтягивает актуальные данные и при необходимости инспекцию.
5. Клиент отображает обогащённую карточку.
6. Улучшенные поля могут быть отправлены обратно в локальную БД.

## API

### Машины

- `GET /api/cars` — список автомобилей с фильтрами, поиском и пагинацией
- `GET /api/cars/:id` — одна карточка автомобиля
- `POST /api/cars` — создание автомобиля
- `PUT /api/cars/:id` — обновление автомобиля
- `DELETE /api/cars/:id` — удаление автомобиля

### Фото

- `POST /api/cars/:id/images` — загрузка изображений
- `DELETE /api/images/:id` — удаление изображения

### Encar

- `GET /api/encar/:encarId` — получение и нормализация данных по машине из Encar

### Админка

- `POST /api/admin/login`
- `GET /api/admin/filter-options`
- `GET /api/admin/stats`

### Парсер

- `GET /api/scraper/status`
- `GET /api/scraper/logs`
- `POST /api/scraper/start`
- `POST /api/scraper/stop`
- `PUT /api/scraper/config`
- `GET /api/scraper/stream`
- `GET /api/scraper/stats`

### Служебный маршрут

- `GET /api/health`

## Локальный запуск

### Требования

- `Node.js >= 18`
- `PostgreSQL`

### 1. Установка

```bash
npm install
```

### 2. Настройка окружения

Скопировать `.env.example` в `.env` и заполнить значения:

```env
DATABASE_URL=postgresql://user:password@localhost:5432/avt_auto
PORT=3001
NODE_ENV=development
ENCAR_PROXY_URL=
CARAPIS_API_KEY=
```

Дополнительно можно задать:

```env
ADMIN_PASSWORD=your-password
FALLBACK_KRW_PER_USD=1485
```

### 3. Запуск API

```bash
npm run server
```

При старте сервер сам применит `server/schema.sql`.

### 4. Запуск фронтенда

В отдельном терминале:

```bash
npm run dev
```

По умолчанию:
- фронт: `http://localhost:5173`
- API: `http://localhost:3001`

В `vite.config.js` уже настроен proxy для `/api` и `/uploads`.

## Сборка production

```bash
npm run build
npm start
```

## Ограничения и важные замечания

### 1. Админ-авторизация простая

Сейчас админ-вход реализован через пароль и упрощённый токен-ответ.
Это удобно для внутреннего инструмента, но не является полноценной enterprise-auth системой.

Для публичного production-доступа стоило бы усилить:
- JWT / session storage;
- rate limiting;
- CSRF / cookie strategy;
- аудит действий администратора.

### 2. Encar может менять структуру

Часть системы зависит от внешней структуры Encar API и HTML-инспекции.
Если Encar изменит:
- DOM страницы;
- названия полей;
- правила выдачи;
- ограничения по сети,

то слой парсинга придётся адаптировать.

### 3. Проект содержит много доменной логики

Большая часть ценности находится не в UI как таковом, а в:
- нормализации данных;
- логике обогащения;
- переводах;
- расчётах;
- парсинге отчётов.

Это важно понимать при развитии проекта: здесь самые чувствительные места находятся в слоях данных и интеграций.
